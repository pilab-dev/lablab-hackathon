# 📑 Phase 3 Technical Spec: Risk, Execution & Verification

*Targeted at Senior Developers building the I/O and Safety Layer of the trading bot.*

In trading, the "Brain" (Phase 2) suggests trades based on sentiment, but the **Risk Layer** is what prevents the bot from blowing up your account, and the **Executor** is the driver that translates those thoughts into real API calls.

---

## 1. The Risk Management Module (`internal/execution/risk.go`)
**Risk Management** is 90% of the game. Its job is to say "NO" to the AI if the AI gets too greedy or if the market is crashing.

### A. Position Sizing (The "How Much" logic)
*   **Concept:** You never bet the whole house on one trade. 
*   **Rule:** Max 10% of total portfolio value per trade.
*   **Implementation:** 
    *   Fetch `TotalValueUSD` from `state.PortfolioState`.
    *   If AI suggests `size_pct: 10.0`, calculate `Amount = (TotalValue * 0.10) / CurrentPrice`.
    *   **Developer Tip:** Always round *down* to the nearest precision allowed by Kraken (e.g., BTC usually allows 8 decimals).

### B. The Circuit Breaker (Daily Loss Limit)
*   **Concept:** If the bot loses too much in 24 hours, it must stop entirely. This prevents "revenge trading" or catching a falling knife.
*   **Rule:** If `CurrentPortfolioValue < (StartingDayValue * 0.95)`, kill all active processes.
*   **Implementation:** Store the `StartingDayValue` at 00:00 UTC. Check every 30 seconds.

### C. Trade Cooldown (Anti-Churn)
*   **Concept:** Prevents the bot from buying and selling the same asset repeatedly in a loop (which wastes money on fees).
*   **Rule:** Min 5 minutes between trades on the same pair.
*   **Implementation:** Maintain a `map[string]time.Time` of the last trade per pair.

## 2. The Trade Executor (`internal/execution/runner.go`)
This is a wrapper around the `kraken-cli`. As a dev, you should treat the CLI as a **stateless JSON microservice**.

### A. Command Wrapper
*   **Function:** `Execute(order TradeOrder) (Result, error)`
*   **CLI Command:** `kraken order buy [pair] [volume] --type market -o json`
*   **Why Market Orders?** For the MVP, we use "Market Orders" because they execute immediately at the best available price. This is vital for a bot that needs to act on its signals right now.

### B. Atomicity & Error Handling
*   **Scenario:** What if the CLI returns an error?
*   **Logic:**
    1.  Log the raw JSON error.
    2.  Check the error code: 
        *   `EOrder:Insufficient funds`: Update local state and pause.
        *   `EAPI:Rate limit exceeded`: Implement exponential backoff.
    3.  Never "assume" a trade happened. Only update your `internal/state` if the CLI returns a `txid` (Transaction ID).

## 3. ERC-8004 Verification (`internal/execution/verify.go`)
This is the "Innovation" part of the hackathon. It proves you didn't just hard-code a 500% profit into your dashboard.

### A. The Performance Payload
Every time a trade finishes, you generate a small JSON object:
```json
{
  "timestamp": 1712074200,
  "pair": "BTCUSD",
  "entry_price": 67250.00,
  "exit_price": 68100.00,
  "pnl_pct": 1.26,
  "reasoning_hash": "sha256(LLM_Reasoning_Text)"
}
```

### B. Cryptographic Signing
*   **Action:** Sign the hash of this payload using a local Private Key (stored in `.env`).
*   **Result:** This signature is stored in InfluxDB alongside the trade. 
*   **Submission:** On the hackathon leaderboard, you provide your Public Key. Anyone can then verify that the trade data you submitted was actually generated by your bot at that time.

## 4. Suggested Implementation Workflow for a Senior Dev

### Step 1: Interface Definition
Define a `Trader` interface. This allows you to swap between `PaperTrader` (dry run) and `LiveTrader` (real money) easily.
```go
type Trader interface {
    GetBalance() (map[string]float64, error)
    PlaceOrder(pair string, side string, volume float64) (string, error)
}
```

### Step 2: The "Safety Wrapper" Pattern
Wrap your `Trader` in a `RiskManager`. This is a classic decorator pattern.
```go
func (rm *RiskManager) PlaceOrder(pair string, side string, volume float64) (string, error) {
    if !rm.PassesCooldown(pair) { return "", errors.New("cooldown") }
    if rm.ExceedsDailyLoss() { return "", errors.New("circuit breaker active") }
    
    return rm.UnderlyingTrader.PlaceOrder(pair, side, volume)
}
```

### Step 3: The CLI Bridge
Use Go's `os/exec` to call `kraken-cli`. 
*   **Set Env Vars:** Pass `KRAKEN_API_KEY` and `KRAKEN_API_SECRET` into the `Cmd.Env`.
*   **Pipe Stderr:** Always capture `stderr` to catch auth or network errors.

### Step 4: Integration with Phase 2
Modify your `main.go` loop:
1.  **Phase 2 Brain** returns a `Decision`.
2.  **Phase 3 RiskManager** validates the `Decision`.
3.  **Phase 3 Executor** calls `kraken-cli`.
4.  **Phase 3 Verifier** signs the result and saves to InfluxDB.

## 📚 Trading Terms Cheat Sheet for Devs:
*   **Pair:** The asset you're trading (BTC/USD).
*   **Side:** Buy (Long) or Sell (Short).
*   **Volume:** The amount of the asset (e.g., 0.001 BTC).
*   **Market Order:** Buy/sell immediately at the current price. (What we use).
*   **Limit Order:** Buy/sell only at a specific price you set.
*   **Slippage:** The difference between the price you *saw* and the price you actually *got* (common in volatile markets).
*   **PnL:** Profit and Loss.
